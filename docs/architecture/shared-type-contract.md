# Shared Type Contract

OpenNivara already uses Specta for backend-to-frontend type generation. Keep this workflow and expand it for the new engine, approval, tool, runtime, and recovery DTOs.

Do not add a new type-generation framework.

## Current Context

Current binding generator:

- `src/bindings.rs` uses `specta::{ts, Type}`.
- Generated output path is `desktop/src/generated/backendTypes.ts`.
- The generated file says it is generated by `cargo test bindings_are_current`.
- `src/bindings.rs` has a `bindings_are_current` test that regenerates the file and fails if stale.

Current issues:

- `SessionSummary` has `source_created`.
- `ToolSecuritySummary` reflects the old security/path model.
- Desktop `AskResponse` is defined locally in `desktop/src-tauri/src/lib.rs` and only includes `session_id` plus `answer`.
- `EngineResponse` is not exported as the shared Desktop contract.

## Canonical Type Rule

Define shared Rust structs in their real backend modules, then export them from `src/bindings.rs`.

Avoid duplicate Desktop-only copies unless absolutely necessary.

Preferred canonical locations:

- engine response types in `src/engine.rs` or `src/engine/types.rs`
- approval view/status types in `src/state/types.rs` or `src/approvals/types.rs`
- operation policy enums in `src/tools/operation_policy.rs`
- tool preview/result display types in `src/tools/types.rs`
- `Surface` and runtime IDs in `src/runtime/types.rs` or `src/runtime/ids.rs`

Desktop Tauri commands should prefer returning root crate types directly:

```rust
ask_opennivara(...) -> Result<opennivara::engine::EngineResponse, String>
```

If a Desktop wrapper remains, it must be structurally identical and exported through Specta.

## Serialization And Naming

Use snake_case fields. The generated TypeScript currently preserves snake_case, so keep it.

Enums should serialize as lowercase snake_case string unions:

```rust
#[serde(rename_all = "snake_case")]
```

Use Specta rename support if needed:

```rust
#[specta(rename_all = "snake_case")]
```

`EngineResponseKind` should generate approximately:

```ts
export type EngineResponseKind = "answer" | "approval_required";
```

## ToolPreviewEnvelope

Use a typed preview envelope instead of raw `preview_json`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Type)]
pub struct ToolPreviewEnvelope {
    pub schema_version: u32,
    pub tool_name: String,
    pub preview_kind: String,
    pub operation_target: Option<String>,
    pub summary: String,
    pub details: serde_json::Value,
}
```

Frontend needs a stable envelope for rendering approval and activity previews. Tool-specific details can remain `serde_json::Value`.

Keep `full_arguments_json` as `Option<serde_json::Value>` because tool args vary by tool.

## SessionSummary Update

Change `SessionSummary` to:

```rust
pub struct SessionSummary {
    pub id: String,
    pub title: String,
    pub updated_at: String,
    pub status: String,
    pub surface_created: String,
    pub actor_id_created: Option<String>,
    pub active: bool,
}
```

`SessionSummary` should no longer export `source_created`.

## ToolSecuritySummary Rename

`ToolSecuritySummary` no longer matches the liberal tool policy.

Recommended eventual rename:

```text
ToolConfigSummary
```

Suggested fields:

- enabled
- max tool rounds
- show tool activity
- allowed roots
- blocked patterns
- max read bytes
- max tool result bytes

Do not rush this rename if it causes large frontend churn, but treat it as the target.

## Binding Export Organization

Add export groups/comments in `src/bindings.rs`:

- Runtime
- Engine
- Approvals
- Tools
- Existing profile/style/preferences/context/theme/memory types

When new types are added:

1. Add derives `Type`, `Serialize`, and `Deserialize`.
2. Add to `src/bindings.rs` export list.
3. Run `cargo test bindings_are_current`.
4. Commit regenerated `desktop/src/generated/backendTypes.ts`.
5. Frontend imports generated types instead of hand-written duplicates.

If Specta emits `any`/`unknown` for `serde_json::Value`, that is acceptable for:

- `details`
- `full_arguments_json`
- `result_json`

Frontend rendering should branch on:

- `preview.preview_kind`
- `approval.status`
- `approval.phase`
- `can_*` action booleans
